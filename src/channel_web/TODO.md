## Prio 1 — 👎👍 Feedback on answers

Implement per-answer thumbs up/down feedback with an optional free-text comment.
The stored trace must be the complete pipeline record — everything sent to and received
from every LLM call and every service call, verbatim.

### Trace schema (stored in Firestore `feedback` collection)

```
trace_id          str    — generated by gateway at pipeline start; returned in answer SSE event
session_id        str
message_id        str    — uuid per turn (already generated in chat.py)
timestamp         str    — ISO 8601, UTC
pipeline_path     str    — "out_of_scope" | "orientation" | "specific" | "broad"

input:
  raw_message     str    — exactly what the user typed
  sanitized_query str    — after sanitize()
  sanitize_warning str | null

classifier:
  corpus_summary  str    — the full corpus outline fed into the prompt
  prompt          str    — the verbatim _PROMPT string as sent to the LLM
  raw_response    str    — response.text from the model (raw JSON string)
  in_scope        bool
  specific_enough bool

rewrite:
  history         list[{q, a}]   — session history slice used as rewrite context
  original_query  str
  rewritten_query str | null     — null if no rewrite was attempted

topics:
  request_query   str            — query sent to POST /topics
  l1_topics       list[{doc_id, id, title}]   — raw response from knowledge service
  topic_count     int            — after sibling consolidation
  labels          list[str]      — consolidated human-readable labels
  overview        bool

knowledge_search:
  request:
    query         str
    overview      bool
  response:
    answer        str            — raw answer from knowledge service (before BROAD_QUERY_PREFIX)
    facts         list[{fact, source_id, valid_at}]
    synthesis_selected_nodes  list[{scoped_id, doc_id, title, ...}]
                               — the selected_nodes from result["synthesis"], verbatim

output:
  answer          str    — final answer shown to user (after BROAD_QUERY_PREFIX prepend if any)
  facts           list[{fact, source_id, valid_at}]

feedback:
  rating          int    — +1 or -1
  comment         str | null
  rated_at        str    — ISO 8601, UTC
```

---

### Architecture

The trace is assembled and stored **by the gateway**, not the channel.

1. Gateway generates a `trace_id` (uuid) at pipeline start.
2. As each pipeline step completes, gateway accumulates the trace dict in memory.
3. After the final `answer` SSE event is queued, gateway writes the trace to
   Firestore `traces/{trace_id}` (fire-and-forget, do not block the response).
4. The `answer` SSE event gains a `trace_id` field alongside `session_id`.

The channel_web `POST /feedback` endpoint:
- Accepts: `{ trace_id, rating: +1|-1, comment: str | null }`
- Updates Firestore `traces/{trace_id}.feedback` with rating + comment + rated_at
- Returns 204

No trace data travels through the browser — the browser only holds `trace_id`.

## Feedback editing

After feedback is submitted, the selected thumb stays highlighted. If the user clicks
either thumb button again, re-open the feedback form pre-filled with their previous
rating and comment, allowing them to change either or both.

On re-submit, overwrite the existing `feedback.*` fields in the Firestore trace doc
(same `POST /feedback` call — the endpoint already does an update, not an insert).
`rated_at` gets a fresh timestamp.

**Browser state**: keep the last submitted `{ rating, comment }` in the same JS object
that holds `traceId`. No persistence needed beyond the current page — if the user
reloads, the highlight is gone, which is acceptable.

## Disclaimer button

The disclaimer button originally worked by submitting a hardcoded question
(`"What are the limitations of this chatbot?"`) as a regular chat query, relying
on the knowledge service to return a canned answer from the corpus.

Since the gateway was introduced, this no longer works: the question is not in
the corpus, so the scope gate classifies it as out-of-scope and returns the
out-of-scope reply instead of a disclaimer.

**Fix options (pick one):**

1. **Hardcoded response (simplest)** — Intercept the disclaimer question in the
   gateway before the scope gate and return a static `DISCLAIMER_TEXT` string
   from `config.py`. No corpus dependency, no LLM call.

2. **Static modal (no backend)** — Replace the submit-question flow entirely.
   Clicking "Disclaimer" opens a modal with hardcoded disclaimer text in the
   frontend. Zero backend involvement. Correct separation of concerns: disclaimer
   is a UI concern, not a knowledge concern.

3. **Corpus entry** — Add a disclaimer document to the corpus so the classifier
   marks it in-scope. Fragile: depends on the classifier and retrieval both
   working correctly for a non-question query.

Option 2 is recommended. The disclaimer text is product copy, not knowledge-base
content — it belongs in the UI layer.
